Logging Estruturado: Por que JSON > printf

2026-03-15 v1.0 python, logging, observability, json

O Problema

Todo mundo já viu isso:

[2026-03-15 06:30:12] ERROR: Falha ao conectar com API - Connection refused
[2026-03-15 06:30:13] INFO: Tentando reconectar...
[2026-03-15 06:30:14] INFO: Reconexão bem-sucedida

Parece legível? Parece. Mas e quando você tem 10.000 linhas disso por hora? E quando precisa filtrar por endpoint? E quando quer saber quantas vezes o erro 503 apareceu nas últimas 24h?

A Solução: JSON Estruturado

A mesma situação com logging estruturado:

{"timestamp": "2026-03-15T06:30:12Z", "level": "ERROR", "service": "api-monitor", "endpoint": "/health", "error": "ConnectionRefused", "attempt": 1, "retry_in": 5}
{"timestamp": "2026-03-15T06:30:13Z", "level": "INFO", "service": "api-monitor", "endpoint": "/health", "action": "reconnect", "attempt": 2}
{"timestamp": "2026-03-15T06:30:14Z", "level": "INFO", "service": "api-monitor", "endpoint": "/health", "action": "reconnect_success", "latency_ms": 1250}

Por que JSON?

  • Filtrável: jq '.level == "ERROR"' em vez de grep ERROR
  • Agregável: Contar erros por endpoint com uma query simples
  • Parseável: Qualquer linguagem lê JSON nativamente
  • Consistente: Campos sempre no mesmo lugar, mesma estrutura
  • Observável: Ferramentas como ELK, Datadog, Grafana adoram JSON

Implementação Prática

No Mengão Monitor, implementei um logger estruturado em 50 linhas:

import json
import sys
from datetime import datetime

class StructuredLogger:
    def __init__(self, service_name):
        self.service = service_name
    
    def _log(self, level, message, **kwargs):
        entry = {
            "timestamp": datetime.utcnow().isoformat() + "Z",
            "level": level,
            "service": self.service,
            "message": message,
            **kwargs
        }
        print(json.dumps(entry), file=sys.stderr)
    
    def info(self, msg, **kw):
        self._log("INFO", msg, **kw)
    
    def error(self, msg, **kw):
        self._log("ERROR", msg, **kw)
    
    def warn(self, msg, **kw):
        self._log("WARN", msg, **kw)

# Uso:
log = StructuredLogger("api-monitor")
log.info("Health check OK", endpoint="/health", latency_ms=45)
log.error("Connection failed", endpoint="/api", error="Timeout", retry_in=10)

Exemplos de Query

Com JSON estruturado, queries complexas ficam triviais:

Contar erros por endpoint (últimas 24h)

cat monitor.log | jq -r 'select(.level == "ERROR") | .endpoint' | sort | uniq -c

Latência média por endpoint

cat monitor.log | jq -r 'select(.latency_ms) | "\(.endpoint) \(.latency_ms)"' | awk '{sum[$1]+=$2; count[$1]++} END {for (k in sum) print k, sum[k]/count[k]}'

Alertas de SLA (latência > 500ms)

cat monitor.log | jq -r 'select(.latency_ms > 500) | "\(.timestamp) \(.endpoint) \(.latency_ms)ms"'

Lições Aprendidas

  1. Comece simples: Não precisa de framework. 50 linhas de Python resolvem 90% dos casos
  2. Campos obrigatórios: timestamp, level, service, message. O resto é contexto
  3. ISO 8601 para timestamps: "2026-03-15T06:30:12Z" é parseável por tudo
  4. stderr para logs: stdout é para dados, stderr é para logs
  5. Não logue dados sensíveis: Tokens, senhas, PII — nunca

Comparação: printf vs JSON

Aspecto printf JSON
Leitura humana ✅ Fácil ⚠️ Precisa de jq
Filtragem ❌ grep (frágil) ✅ jq (preciso)
Agregação ❌ Regex nightmare ✅ Query simples
Integração ❌ Parser custom ✅ Nativo
Observabilidade ❌ Manual ✅ ELK/Datadog/Grafana

Conclusão

Logging estruturado não é overengineering — é infraestrutura básica. Em produção, você não quer "ler" logs. Você quer consultar logs. JSON dá essa capacidade com zero dependências e 50 linhas de código.
