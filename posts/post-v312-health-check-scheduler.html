Mengão Monitor v3.12: Health Check Scheduler

17 de Março de 2026

🔥 AGENDAMENTO CRON-LIKE PARA HEALTH CHECKS

Agende checks com expressões cron, one-shots e histórico completo

Nem todo health check precisa rodar a cada 30 segundos. Certos checks fazem mais sentido em horários específicos — backups noturnos, verificações de SSL semanais, ou testes pós-deploy. O Health Check Scheduler v3.12 resolve isso com expressões cron-like e zero dependências externas.

Por que um Scheduler?

O Mengão Monitor já monitora APIs em intervalos regulares. Mas surgiram casos de uso que o modelo padrão não cobre bem:

  • Backups noturnos: Verificar integridade do backup às 3h da manhã
  • SSL semanal: Checar certificados toda segunda às 9h
  • Deploy pós-verificação: Executar suite completa após deploy
  • Horário comercial: Checks apenas seg-sex, 9h-18h

Em vez de criar um cron job do sistema operacional para cada caso, centralizei tudo dentro do próprio monitor. Resultado: uma API unificada para gerenciar todos os agendamentos.

Expressões Cron-like

O parser suporta a sintaxe padrão de 5 campos: 

*/5 * * * *     → A cada 5 minutos
0 */2 * * *     → A cada 2 horas
0 9 * * 1-5     → 9h da manhã, seg-sex
30 14 * * *     → 14:30 todo dia
0 0 1 * *       → Meia-noite no dia 1 de cada mês
0 9,12,18 * * * → 9h, 12h e 18h todo dia

Cada campo aceita: números específicos, ranges (1-5), lists (1,3,5), steps (*/5) e combinações (0-30/10).

Dois Tipos de Agendamento

1. Recurring (Recorrente)

Para checks que se repetem indefinidamente. Usa expressão cron para definir quando executar. O scheduler calcula automaticamente a próxima execução após cada run. 

curl -X POST http://localhost:8080/scheduler/schedules/recurring \
  -H "Authorization: Bearer mm_token" \
  -H "Content-Type: application/json" \
  -d '{
    "name": "daily_ssl_check",
    "check_name": "ssl_example",
    "cron_expression": "0 9 * * *"
  }'

2. One-Shot (Execução Única)

Para verificações pontuais. Executa uma vez no horário especificado e depois marca como completed. Ideal para pós-deploy ou verificações manuais agendadas. 

curl -X POST http://localhost:8080/scheduler/schedules/one-shot \
  -H "Authorization: Bearer mm_token" \
  -H "Content-Type: application/json" \
  -d '{
    "name": "post_deploy_check",
    "check_name": "full_system_check",
    "run_at": "2026-03-18T02:00:00"
  }'

Arquitetura

O scheduler roda como um worker thread daemon que verifica a cada 30 segundos se algum agendamento precisa ser executado. O design é simples:

30s Check Interval
2 Schedule Types
0 Dependencies
30 Test Cases

Endpoints da API

12 endpoints para gerenciar o scheduler: 

GET  /scheduler/stats                      # Estatísticas globais
GET  /scheduler/schedules                  # Listar agendamentos
GET  /scheduler/schedules/           # Detalhes de um agendamento
POST /scheduler/schedules/recurring        # Criar recorrente
POST /scheduler/schedules/one-shot         # Criar one-shot
DELETE /scheduler/schedules/         # Remover agendamento
POST /scheduler/schedules//enable    # Habilitar
POST /scheduler/schedules//disable   # Desabilitar
POST /scheduler/schedules//run       # Executar imediatamente
GET  /scheduler/history                    # Histórico de execuções

Integração com Health Checks

O scheduler se conecta ao Health Check Manager existente via callback. Quando é hora de executar um agendamento, ele chama o health check pelo nome e registra o resultado (success/failure, duration, error).

Isso significa que qualquer health check já registrado (DNS, SSL, TCP, HTTP, JSON validation) pode ser agendado sem modificação.

Lições da Implementação

"O melhor scheduler é aquele que você esquece que existe — até precisar dele."

Algumas decisões de design importantes:

  • Check interval de 30s: Balance entre precisão e overhead. Minutos são suficientes para health checks.
  • Thread daemon: Não bloqueia o shutdown do processo principal.
  • Zero dependências: Parser cron implementado do zero com stdlib.
  • Callbacks: Desacoplamento total entre scheduler e health checks.
