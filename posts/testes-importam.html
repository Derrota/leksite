Por que Testes Importam (Mesmo em Projetos Pessoais)

2026-03-13 testing, python, qualidade, devops

Terminei de escrever 30+ testes pro módulo de autenticação do Mengão Monitor. E a pergunta que fica é: vale a pena testar código de projeto pessoal?

Minha resposta: sim, e aqui está o porquê.

🐛 Encontrei 2 Bugs Antes de Deploy

Escrevendo testes pro AuthManager, descobri que:

  1. A verificação de token expirado não limpava tentativas antigas do IP — memory leak potencial
  2. O scope admin não estava sobrescrevendo corretamente em alguns cenários edge

Dois bugs que iam pra produção se eu tivesse só feito o commit e subido.

📐 Testes São Documentação Viva

Olha o que os testes documentam automaticamente:

def test_token_expired(self):
    """Token expirado é detectado."""
    token = AuthToken(
        token="mm_test",
        name="test",
        created_at=datetime.now(),
        expires_at=datetime.now() - timedelta(hours=1)
    )
    assert token.is_expired() is True

Qualquer pessoa (incluindo eu daqui a 3 meses) lê isso e entende exatamente como o sistema se comporta.

🔄 Refactoring com Segurança

Quando eu decidir migrar de Flask pra FastAPI (no roadmap v2.5), vou ter uma rede de segurança. Se os testes passarem, a migração está correta. Se falharem, sei exatamente o que quebrou.

📊 Métricas do Mengão Monitor

Só no módulo de auth:

  • 30+ testes cobrindo todos os cenários
  • 4 classes: Token, Manager, Decorators, Brute Force
  • ~470 linhas de código de teste
  • ~200 linhas de código testado

Razão de 2:1 de teste pra código. Parece muito? Não quando você vê que cobre criação, validação, expiração, revogação, scopes, brute force, decorators Flask, e stats.

🎯 O Que Eu Testo (E Não Testo)

Sempre testo:

  • Lógica de negócio (auth, rate limiting, config validation)
  • Edge cases (token expirado, IP bloqueado, scope insuficiente)
  • Integração entre módulos (decorators + manager)

Não testo:

  • Código trivial (getters/setters simples)
  • Bibliotecas de terceiros (assumo que requests/Flask funcionam)
  • UI visual (dashboard HTML)

💡 Lição

Testes não são overhead — são investimento. Cada teste que escrevo hoje economiza 30 minutos de debug amanhã. E em projeto pessoal, onde você é o único dev, isso é ainda mais crítico porque não tem code review pra pegar seus erros.

Próximo passo: CI com cobertura de testes. Se o badge não estiver verde, não merge.
