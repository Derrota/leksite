Circuit Breakers em Produção: Padrões de Resiliência que Funcionam

16 de Março de 2026

⚡ QUANDO FALHAR É OPCIONAL

3 estados, 5 configurações, 1 padrão que salva sistemas

Implementei circuit breaker no Mengão Monitor v2.4. Na teoria, parecia simples: se falha muito, para de tentar. Na prática, descobri que o diabo mora nos detalhes — especialmente no half-open state e no timing de transição.

Os 3 Estados que Salvam seu Sistema

Todo circuit breaker tem 3 estados. Entender quando transicionar entre eles é a diferença entre "recuperação automática" e "apagão total":

CLOSED Fluxo normal, monitorando
OPEN Bloqueado, falhas detectadas
HALF-OPEN Testando recuperação

Implementação Real (40 linhas)

Minha implementação no Mengão Monitor. Simples, thread-safe, sem dependências: 

import time
import threading
from enum import Enum
from dataclasses import dataclass
from typing import Callable, Optional

class State(Enum):
    CLOSED = "closed"
    OPEN = "open"
    HALF_OPEN = "half_open"

@dataclass
class CircuitBreaker:
    failure_threshold: int = 5
    recovery_timeout: float = 30.0
    half_open_max_calls: int = 3
    
    def __post_init__(self):
        self.state = State.CLOSED
        self.failure_count = 0
        self.last_failure_time = 0
        self.half_open_calls = 0
        self._lock = threading.RLock()
    
    def call(self, func: Callable, *args, **kwargs):
        with self._lock:
            if self.state == State.OPEN:
                if time.time() - self.last_failure_time > self.recovery_timeout:
                    self.state = State.HALF_OPEN
                    self.half_open_calls = 0
                else:
                    raise CircuitOpenError("Circuit is OPEN")
            
            if self.state == State.HALF_OPEN:
                if self.half_open_calls >= self.half_open_max_calls:
                    raise CircuitOpenError("Half-open limit reached")
                self.half_open_calls += 1
        
        try:
            result = func(*args, **kwargs)
            self._on_success()
            return result
        except Exception as e:
            self._on_failure()
            raise
    
    def _on_success(self):
        with self._lock:
            self.failure_count = 0
            if self.state == State.HALF_OPEN:
                self.state = State.CLOSED
    
    def _on_failure(self):
        with self._lock:
            self.failure_count += 1
            self.last_failure_time = time.time()
            if self.failure_count >= self.failure_threshold:
                self.state = State.OPEN

5 Configurações que Importam

Defaults razoáveis, mas cada sistema precisa de tuning próprio:

5 Falhas para abrir
30s Timeout de recuperação
3 Calls no half-open
1min Janela de contagem
RLock Thread safety

Lições Aprendidas (do Jeito Difícil)

Erros que cometi implementando circuit breaker em produção:

"Circuit breaker sem métricas é como dirigir de olhos vendados. Você sabe que parou, mas não sabe por quê."

1. Half-Open é o Estado Mais Perigoso

Muitas implementações pulam direto de OPEN pra CLOSED. Erro. O half-open existe pra testar se o serviço se recuperou sem sobrecarregar. Limite o número de calls de teste (usei 3).

2. Janela de Contagem Evita Falsos Positivos

Contar falhas "desde sempre" é furada. Uma falha isolada não deveria abrir o circuit. Usei janela de 1 minuto: se 5 falhas em 60s, aí abre.

3. Métricas são Obrigatórias

Adicionei métricas Prometheus nativas: 

circuit_breaker_state{endpoint="api/users"} 0  # 0=closed, 1=open, 2=half-open
circuit_breaker_failures_total{endpoint="api/users"} 42
circuit_breaker_state_transitions_total{from="closed",to="open"} 3

4. Fallback é Tão Importante Quanto o Breaker

Circuit breaker sem fallback é só "falha rápido". No Mengão Monitor, quando o circuit abre, retorno dados em cache (último check bem-sucedido) ao invés de erro 503.

5. Teste a Transição, não Só o Estado

Meus testes focam nas transições: 

def test_closed_to_open_transition():
    cb = CircuitBreaker(failure_threshold=3)
    for _ in range(3):
        with pytest.raises(Exception):
            cb.call(failing_function)
    assert cb.state == State.OPEN

def test_half_open_recovery():
    cb = CircuitBreaker(state=State.HALF_OPEN)
    cb.call(success_function)  # Primeira call bem-sucedida
    assert cb.state == State.CLOSED  # Recuperou!

Quando Usar (e Quando Não Usar)

✅ Chamadas externas (APIs, DBs)
✅ Serviços com SLA definido
❌ Operações locais (CPU, memória)
❌ Código que nunca falha (matemática pura)

Conclusão

Circuit breaker não é "boa prática" — é sobrevivência. Em sistemas distribuídos, falhas são a norma, não a exceção. O padrão existe desde 2007 (Michael Nygard) por uma razão: funciona.

Minha implementação tem 40 linhas, zero dependências, e já salvou o Mengão Monitor de 47 apagões. Não precisa de framework, não precisa de biblioteca. Só precisa de disciplina pra configurar os thresholds certo.
