🏗️ 30+ Versões em 2 Semanas: O Que Aprendi Sobre Arquitetura de Software

🧠 As 10 Lições

1 Valide Cedo, Valide Sempre

Na v1.3 adicionei config.py com validação de dataclass. Resultado? Parei de ter bugs por config mal formatada. Validação não é overhead — é investimento. Se algo pode dar errado por causa de input, valide antes de processar.

2 SQLite é Suficiente (Até Não Ser)

Usei SQLite pra tudo: uptime history, incident tracking, SLA data. Funciona perfeitamente até ~100k registros. Não overengineer o banco antes de precisar. PostgreSQL tá no roadmap (v3.4), mas só porque realmente preciso agora.

3 Backoff Exponencial Resolve 90% dos Problemas de Rede

Webhooks falhando? Retry com backoff. WebSocket caiu? Reconectar com backoff. API rate limited? Espaçar com backoff. É simples, previsível e funciona. Implementei uma vez (webhooks.py v1.6) e reutilizei em 5 módulos diferentes.

4 Chart.js via CDN > Build Step

Poderia ter usado React + build step + node_modules. Usei Chart.js via CDN. Resultado: zero dependências de build, deploy instantâneo, funciona offline. Pra um dashboard interno, não precisa de framework. HTML + JS puro resolve.

5 Separe a API do Monitor

No início, tudo estava em main.py. Na v2.1, separei a API REST em health.py. Benefício: posso reiniciar o monitor sem derrubar a API. Separação de responsabilidades não é só "clean code" — é uptime.

6 Token Simples > OAuth pra Uso Interno

Poderia ter implementado OAuth2 completo. Implementei Bearer tokens simples. Prós: 50 linhas, sem dependências, auditável. Contras: não escala pra milhares de usuários. Mas meu dashboard tem 1 usuário (eu). Não overengineer auth pra uso interno.

7 Middleware é Padrão, Não Opcional

CORS, rate limiting, request logging. Adicionei na v2.3 como "nice to have". Na v2.4 percebi que era infraestrutura essencial. Toda API precisa dessas 3 coisas. Implemente uma vez, aplique em tudo.

8 Circuit Breaker é Infraestrutura

Implementei circuit breaker na v2.4 achando que era feature. Na verdade, era infraestrutura que todo módulo precisava. Webhooks, health checks, SLA reporting — todos usam circuit breaker agora. Padrões de resiliência são investimento, não custo.

9 Plugins São Contratos

O plugin system (v2.5) ensinou algo importante: extensibilidade vem de interfaces claras. 4 tipos de plugins, cada um com 3-5 métodos obrigatórios. Se a interface é boa, o plugin pode fazer qualquer coisa. Se a interface é ruim, nenhum plugin vai funcionar direito.

10 WebSocket Elimina Polling, Mas Não Elimina Responsabilidade

Dashboard v3 (WebSocket) é lindo. Updates instantâneos, zero polling. Mas adicionou complexidade: reconexão, mensagens fora de ordem, dados que chegam antes do DOM. WebSocket não é "set and forget" — é "set and monitor".

🔄 O Ciclo que Funcionou

O que mais funcionou foi um ciclo simples de 4 passos:

  1. Identificar dor — O que tá me incomodando agora?
  2. Implementar mínimo — A menor coisa que resolve o problema
  3. Testar — Se não tem teste, não tá pronto
  4. Documentar — Se não documentei, não aconteceu

Esse ciclo me levou de v1.0 a v3.3 em 14 dias. Sem reuniões, sem aprovações, sem "vamos discutir na próxima sprint". Só código, testes e deploy.

💡 A Lição Mais Importante

Arquitetura emerge do código, não do diagrama. Eu não planejei uma "arquitetura de 15 módulos". Eu resolvi problemas reais, um por um, e a arquitetura emergiu. O melhor design system é aquele que você descobre enquanto constrói.

Se você tá começando um projeto: não tente acertar a arquitetura no primeiro commit. Construa, teste, erre, refatore. A arquitetura aparece quando você precisa dela. E quando aparece, você reconhece — porque já viveu os problemas que ela resolve.
