Mengão Monitor v1.6: Rate Limiting e Retry Automático

Projeto 📅 2026-03-13 ⏱️ 5 min de leitura

Depois do v1.5 com métricas de sistema, o Mengão Monitor ganhou duas features que tavam faltando pra produção real: rate limiting e retry automático. Porque alerta que não chega é alerta inútil, e alerta que spamma é pior que nenhum alerta.

O Problema

Imagina: sua API caiu. O monitor detecta e manda webhook pro Discord. Mas a rede tá instável — o webhook falha. E aí? Você perdeu o alerta. Ou pior: a API fica flipando (up/down/up/down) e você recebe 50 alertas em 5 minutos. Seu canal vira um inferno de notificações.

⚠️ Lições aprendidas na marra: Webhook sem retry = alerta perdido. Webhook sem rate limit = spam garantido.

Rate Limiting (v1.6)

Novo módulo rate_limiter.py com proteção em múltiplas camadas:

Camada Limite Propósito
Por minuto 5 alertas Evita rajadas instantâneas
Por hora 30 alertas Controla volume horário
Por dia 100 alertas Limite diário absoluto
Burst 3 em 60s Proteção contra flip-flop
Cooldown 5 min/endpoint Evita repetição do mesmo alerta

Como Funciona

from rate_limiter import RateLimiter, RateLimitConfig

# Configuração customizada
config = RateLimitConfig(
    max_alerts_per_minute=5,
    max_alerts_per_hour=30,
    burst_limit=3,
    cooldown_seconds=300
)

limiter = RateLimiter(config)

# Verifica se pode enviar alerta
if limiter.allow_alert("minha-api"):
    # Envia webhook
    webhook_sender.send(result)
else:
    # Bloqueado por rate limit
    logger.warning("Rate limit atingido")

Thread-Safe

O rate limiter usa threading.Lock pra ser seguro em ambientes concorrentes. Se você tiver múltiplas threads checando APIs, o contador não vai ter race conditions.

Retry Automático (v1.6)

O webhooks.py foi reescrito com retry automático e backoff exponencial:

# Fluxo de retry
Tentativa 1 → Falha (timeout)
  ↓ espera 1s
Tentativa 2 → Falha (500)
  ↓ espera 3s  
Tentativa 3 → Falha (connection error)
  ↓ erro final reportado

Configurações

  • 3 tentativas por webhook
  • Backoff: 1s → 3s → 5s
  • Timeout: 10s por request
  • 4xx: Não retry (erro do cliente)
  • 5xx: Retry automático (erro do servidor)

WebhookResult

Novo dataclass que retorna detalhes de cada envio:

@dataclass
class WebhookResult:
    success: bool
    platform: str      # "discord", "slack", "telegram"
    status_code: int   # HTTP status
    error: str         # mensagem de erro se falhou
    attempts: int      # quantas tentativas foram feitas

Integração

Os dois módulos trabalham juntos no fluxo principal:

  1. API check detecta status change
  2. Rate limiter verifica se pode alertar
  3. Se permitido, webhooks enviam com retry
  4. Stats são atualizadas (sent, failed, retries, rate_limited)
