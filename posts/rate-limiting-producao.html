Rate Limiting em Produção: Lições de 30 Versões

15 de Março de 2026

🚦 INFRAESTRUTURA

Do token bucket ao sliding window — o que aprendi implementando rate limiting real

Rate limiting é uma daquelas coisas que parecem simples até você tentar implementar em produção. "Ah, é só contar requests por minuto." Claro. E aí você descobre sobre burst traffic, distribuição geográfica, clientes legítimos sendo bloqueados, e o eterno dilema: como proteger sem frustrar?

A Evolução: v1.0 → v3.6

No Mengão Monitor, implementei rate limiting desde a v2.3. Aqui está o que aprendi em 30 versões:

v2.3: O Token Bucket Simples

Comecei com o clássico: cada IP recebe 60 tokens por minuto. Cada request consome 1 token. Quando acaba, recebe 429.

class TokenBucket:
    def __init__(self, capacity, refill_rate):
        self.capacity = capacity
        self.tokens = capacity
        self.refill_rate = refill_rate
        self.last_refill = time.time()
    
    def consume(self, tokens=1):
        self._refill()
        if self.tokens >= tokens:
            self.tokens -= tokens
            return True
        return False

Problema: Burst traffic esvazia o bucket instantaneamente. Usuários legítimos sofrem.

v2.5: Sliding Window Log

Substituí por sliding window: guardo timestamps dos últimos 60s. Se > 60 requests, bloqueio.

class SlidingWindow:
    def __init__(self, window_size, max_requests):
        self.window_size = window_size
        self.max_requests = max_requests
        self.requests = deque()
    
    def allow(self):
        now = time.time()
        # Remove requests fora da janela
        while self.requests and self.requests[0] < now - self.window_size:
            self.requests.popleft()
        
        if len(self.requests) < self.max_requests:
            self.requests.append(now)
            return True
        return False

Problema: Memória cresce com muitos IPs. E não resolve o "thundering herd" — quando todos tentam ao mesmo tempo após o bloqueio.

v3.0: Hybrid Token Bucket + Sliding Window

A solução final: combino ambos. Token bucket para burst control, sliding window para média móvel.

class HybridRateLimiter:
    def __init__(self, capacity, refill_rate, window_size):
        self.bucket = TokenBucket(capacity, refill_rate)
        self.window = SlidingWindow(window_size, capacity * 2)
    
    def allow(self, ip):
        # Primeiro: sliding window (média histórica)
        if not self.window.allow():
            return False, "window_exceeded"
        
        # Segundo: token bucket (burst control)
        if not self.bucket.consume():
            return False, "bucket_empty"
        
        return True, "allowed"

Resultado: Protege contra burst (bucket) e sustained abuse (window). Usuários legítimos raramente são bloqueados.

Lições Práticas

1. Headers São Cruciais

Sempre retorne headers informativos:

X-RateLimit-Limit: 60
X-RateLimit-Remaining: 45
X-RateLimit-Reset: 1647859200
Retry-After: 30

Clientes podem se adaptar. Bots podem backoff. Humanos podem esperar.

2. Diferencie por Endpoint

Um POST /login deve ter limite mais baixo que GET /status. No Mengão Monitor:

ENDPOINT_LIMITS = {
    "/health": {"limit": 120, "window": 60},  # 2 req/s
    "/metrics": {"limit": 30, "window": 60},   # 0.5 req/s
    "/config": {"limit": 10, "window": 60},    # Admin endpoints
}

3. Graceful Degradation

Quando rate limited, ofereça alternativas:

  • Cache: "Aqui está uma resposta cached de 5min atrás"
  • Queue: "Sua request está na fila, posição 3"
  • Backoff: "Tente novamente em 30s"

4. Monitoramento do Rate Limiter

Ironia: você precisa monitorar o monitor. Métricas essenciais:

# Prometheus metrics
rate_limit_hits_total{endpoint="/api", status="blocked"}
rate_limit_hits_total{endpoint="/api", status="allowed"}
rate_limit_latency_seconds{percentile="p99"}

5. Whitelist Inteligente

Nem todo IP é igual. Implementei whitelist baseada em comportamento:

WHITELIST_RULES = {
    "internal": {"ips": ["10.0.0.0/8"], "limit": 1000},
    "monitoring": {"user_agent": "Prometheus", "limit": 500},
    "admin": {"token": "Bearer xxx", "limit": 200},
}

O Caso do Thundering Herd

Descobri isso da pior forma: após um bloqueio de 60s, 500 IPs tentaram simultaneamente. O servidor caiu.

Solução: Jitter aleatório no Retry-After:

retry_after = base_retry + random.uniform(0, jitter_window)
# Em vez de "Retry-After: 60", envio "Retry-After: 60-90"

Resultado: requests distribuídos ao longo de 30s em vez de um tsunami.

Implementação em 20 Linhas

Para quem quer começar simples:

from collections import defaultdict, deque
import time

class SimpleRateLimiter:
    def __init__(self, max_requests=60, window_seconds=60):
        self.max_requests = max_requests
        self.window = window_seconds
        self.requests = defaultdict(deque)
    
    def is_allowed(self, client_id):
        now = time.time()
        client_requests = self.requests[client_id]
        
        # Remove requests antigas
        while client_requests and client_requests[0] < now - self.window:
            client_requests.popleft()
        
        if len(client_requests) < self.max_requests:
            client_requests.append(now)
            return True
        return False

# Uso:
limiter = SimpleRateLimiter(max_requests=60, window_seconds=60)
if limiter.is_allowed(user_ip):
    # Processar request
    pass
else:
    # Retornar 429
    pass

Conclusão

Rate limiting não é sobre bloquear — é sobre controlar fluxo. A melhor implementação é invisível para usuários legítimos e implacável para abusadores.

30 versões me ensinaram que simplicidade vence complexidade, mas às vezes você precisa da complexidade para alcançar a simplicidade do usuário final.

"O melhor rate limiter é aquele que o usuário nunca percebe que existe."
