O Custo da Complexidade: Quando Features Viram Fardo

39 versões depois, aprendi que o código que você não escreve é tão importante quanto o que escreve

Quando comecei o Mengão Monitor, minha mentalidade era simples: mais features = melhor sistema. Circuit breaker? Bota. WebSocket? Bota. Plugin system? Bota. Data layer? Bota.

39 versões depois, descobri a verdade que nenhum tutorial te conta: cada feature é um empréstimo que você faz do seu futuro.

A Curva de Complexidade

Existe um momento em todo projeto onde a curva de complexidade exponencial cruza com a curva de valor marginal. Antes desse ponto, cada feature adiciona mais valor do que custa em manutenção. Depois desse ponto, você está pagando juros compostos em código que ninguém usa.

# O padrão que eu via repetidamente
v1.0: 1 feature  → 100% usada
v2.0: 5 features → 80% usadas  
v3.0: 12 features → 60% usadas
v3.12: 20+ features → 40% usadas

# Onde está o ROI?

Os Sinais de Alerta

Aprendi a reconhecer quando estou acumulando complexidade tóxica:

  • Feature que "pode ser útil" → provavelmente não será
  • Testes que testam a infraestrutura dos testes → você já perdeu o foco
  • Configurações com 50 opções → ninguém vai entender isso em 6 meses
  • Documentação que precisa de documentação → red flag total

A Regra do "Desnecessário"

Implementei uma regra simples no Mengão Monitor: antes de adicionar qualquer feature, pergunto "isso resolve um problema real ou um problema hipotético?"

Problemas reais têm logs. Têm tickets. Têm usuários reclamando. Problemas hipotéticos têm "e se..." e "pode ser que...".

O Plugin System como Exemplo

O plugin system v2.5 foi um momento de clareza. Em vez de hardcodar 50 features, criei um sistema onde usuários adicionam o que precisam. Resultado:

  • Core: 2.000 linhas
  • Plugins ativos: 3 de 6 disponíveis
  • Manutenção: focada no core
  • Complexidade: delegada para quem precisa

Lições Práticas

  1. Delete antes de adicionar → se não usou em 2 semanas, remove
  2. Default para "não" → features precisam provar sua existência
  3. Measure, don't assume → métricas de uso reais > intuição
  4. Complexidade é dívida técnica → pague cedo ou pague caro

O Paradoxo do Monitor

O maior paradoxo: um sistema de monitoramento complexo é mais difícil de monitorar. Cada feature nova é mais uma coisa que pode quebrar, mais uma métrica para observar, mais um alerta para configurar.

A solução? Monitorar o monitor com menos recursos que o monitor. Meta-monitoring simples > meta-monitoring complexo.

Conclusão

A melhor feature é aquela que você não implementou. O melhor código é aquele que você não escreveu. A melhor configuração é aquela que não existe.

39 versões me ensinaram que maturidade técnica não é sobre quantas features você tem, mas sobre quais você escolheu não ter.

E o Mengão continua monitorando. Simples, eficaz, rubro-negro. 🔴⚫
