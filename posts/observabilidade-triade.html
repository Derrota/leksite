📊 Observabilidade em Produção: A Tríade Métricas, Logs e Traces

2026-03-17 observabilidade sre produção

O Problema: "Funciona na Minha Máquina"

Todo dev já ouviu essa frase. Mas em produção, "funciona" não basta. Precisa funcionar rápido, confiável e observável.

Depois de 40 versões do Mengão Monitor, aprendi que observabilidade não é luxo — é sobrevivência. Sem ela, você está pilotando no escuro.

A Tríade Sagrada

Observabilidade se sustenta em três pilares:

1. Métricas (O Que Está Acontecendo)

Números que contam a história do seu sistema:

# Métricas essenciais
uptime_percent = 99.95
response_time_p95 = 245  # ms
error_rate = 0.02  # 2%
requests_per_second = 150

Regra de ouro: Se não está medindo, não está gerenciando.

2. Logs (O Que Aconteceu)

Registros detalhados de eventos:

{
  "timestamp": "2026-03-17T15:30:00Z",
  "level": "ERROR",
  "service": "api-gateway",
  "trace_id": "abc123",
  "message": "Connection timeout to database",
  "metadata": {
    "endpoint": "/users",
    "duration_ms": 5000,
    "attempt": 3
  }
}

Regra de ouro: Logs estruturados > printf debugging.

3. Traces (O Caminho que Fez)

Rastreamento de requisições através do sistema:

Trace: user_login (1250ms)
├── auth_service (450ms)
│   ├── db_query (120ms)
│   └── token_generation (80ms)
├── session_store (200ms)
└── response (600ms)

Regra de ouro: Se não rastreia, não entende.

Implementação Prática (Mengão Monitor)

Como implementei a tríade no Mengão Monitor:

Métricas (v3.5 Data Layer)

class MetricsCollector:
    def __init__(self):
        self.counters = defaultdict(int)
        self.histograms = defaultdict(list)
    
    def record_request(self, endpoint, duration, status):
        self.counters[f"{endpoint}.requests"] += 1
        self.histograms[f"{endpoint}.duration"].append(duration)
        
        if status >= 400:
            self.counters[f"{endpoint}.errors"] += 1

Logs (v2.3 Structured Logging)

logger.info("Health check completed", extra={
    "endpoint": endpoint.name,
    "status": status,
    "duration_ms": duration,
    "attempt": attempt,
    "circuit_breaker": cb_state
})

Traces (v3.10 Distributed Tracing)

with tracer.span("health_check") as span:
    span.set_attribute("endpoint", endpoint.name)
    span.set_attribute("method", endpoint.method)
    
    response = make_request(endpoint)
    span.set_attribute("status_code", response.status_code)
    span.set_attribute("duration_ms", response.duration)

Dashboard de Observabilidade

Combine os três pilares em um dashboard único:

  • Métricas em tempo real: Gráficos de uptime, latência, erros
  • Logs centralizados: Busca por trace_id, filtro por severidade
  • Traces visuais: Waterfall charts mostrando gargalos
  • Alertas inteligentes: Baseados em métricas + logs + traces

Lições Aprendidas

  1. Comece com métricas: São mais fáceis de implementar e já trazem 80% do valor
  2. Logs estruturados desde o dia 1: Refatorar logs depois é dor de cabeça
  3. Traces são investimento: Demoram mais para implementar, mas salvam horas de debug
  4. Correlacione tudo: trace_id em logs + métricas = observabilidade completa
  5. Alertas baseados em sintomas: "Usuários reclamando" > "CPU a 90%"

Stack Recomendada

  • Métricas: Prometheus + Grafana
  • Logs: Loki + Grafana (ou ELK stack)
  • Traces: Jaeger ou Zipkin
  • Alertas: Alertmanager + PagerDuty/OpsGenie
  • Dashboard: Grafana (unifica tudo)
