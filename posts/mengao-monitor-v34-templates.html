📋 Mengão Monitor v3.4: Health Check Templates

v3.4 14 Mar 2026 • 8 min leitura

Depois de 33 versões construindo o Mengão Monitor, percebi um padrão: toda vez que precisava validar uma API, eu reescrevia a mesma lógica de assertions. Status code, response time, JSON schema... sempre igual.

O v3.4 resolve isso com Health Check Templates — templates pré-definidos que encapsulam a lógica comum de validação de APIs.

O Problema

Antes do v3.4, validar uma API significava:

# Código repetitivo
response = requests.get(url)
assert response.status_code == 200
assert response.elapsed.total_seconds() < 2.0
data = response.json()
assert "status" in data
assert data["status"] == "ok"
# ... mais 20 linhas de validação

Todo endpoint tinha seu próprio código de validação. Sem reutilização, sem padrão, sem histórico.

A Solução: Templates

Agora, validar uma API REST é uma linha:

from health_check_templates import RESTTemplate

template = RESTTemplate(
    name="my-api",
    url="https://api.example.com/health",
    expected_status=200,
    max_response_time=2.0,
    json_schema={"status": str, "version": str},
    required_headers=["X-Request-Id"]
)

result = template.check()
print(result.passed, result.response_time)

8 Tipos de Assertions

STATUS_CODE - HTTP status
RESPONSE_TIME - Latência
JSON_SCHEMA - Estrutura
JSON_PATH - Valor em path
HEADER - Headers HTTP
BODY_CONTAINS - Texto no body
BODY_REGEX - Regex no body
CUSTOM - Callable

Cada assertion tem severity level: INFO, WARNING, ERROR, CRITICAL. Assertions CRITICAL falham o check inteiro. WARNING só avisa.

GraphQL Template

GraphQL tem suas particularidades. O template lida com query validation, error detection e data paths:

from health_check_templates import GraphQLTemplate

template = GraphQLTemplate(
    name="users-api",
    url="https://api.example.com/graphql",
    query="{ users { id name email } }",
    max_response_time=3.0,
    expect_no_errors=True,
    data_paths=[
        {"path": "users.0.id", "operator": "gt", "value": 0}
    ]
)

TemplateChecker

O TemplateChecker gerencia múltiplos templates com histórico e uptime:

checker = TemplateChecker()
checker.register("api1", template1)
checker.register("api2", template2)

# Executar todos
results = checker.check_all()

# Histórico
history = checker.get_history("api1", limit=10)

# Uptime
uptime = checker.get_uptime("api1")  # 99.95%

# Stats globais
stats = checker.get_stats()
# {"total_checks": 142, "passed": 140, "avg_response_time": 0.234}

Presets Prontos

Templates pré-configurados para stacks comuns:

from health_check_templates import (
    create_kubernetes_healthz,
    create_elasticsearch_health,
    create_postgres_health
)

# Kubernetes
checker.register("k8s", create_kubernetes_healthz(
    "http://localhost:8080/healthz"
))

# Elasticsearch
checker.register("es", create_elasticsearch_health(
    "http://localhost:9200"
))
Zero dependências: O módulo inteiro usa apenas stdlib. requests é opcional — se não tiver, retorna erro graceful em vez de crash.

JSON Schema Validation

A validação de JSON schema é built-in, sem precisar de jsonschema lib:

json_schema = {
    "status": str,
    "version": str,
    "data": {
        "count": int,
        "items": [{"id": int, "name": str}]
    }
}

template = RESTTemplate(
    name="api",
    url="https://api.example.com",
    json_schema=json_schema
)

Valida tipos, chaves obrigatórias, objetos aninhados e arrays.

JSON Path Operators

Para validação de valores específicos:

json_paths = [
    {"path": "data.count", "operator": "gt", "value": 0},
    {"path": "status", "operator": "eq", "value": "healthy"},
    {"path": "message", "operator": "contains", "value": "ok"},
    {"path": "version", "operator": "regex", "value": r"\d+\.\d+"}
]

Operadores: eq, ne, gt, lt, gte, lte, contains, regex.

Lição: Composição > Configuração

O design do v3.4 seguiu o princípio de composição. Em vez de um monólito configurável, temos:

  • Assertion - Unidade básica de validação
  • BaseTemplate - Template com lista de assertions
  • RESTTemplate / GraphQLTemplate - Especializações
  • TemplateChecker - Gerenciador com histórico

Cada camada adiciona funcionalidade sem complicar as anteriores. Assertions podem ser reutilizadas entre templates. Templates podem ser combinados no checker.
