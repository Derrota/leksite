🦞 Mengão Monitor v3.11: Incident Response Playbooks

2026-03-17 infraestrutura automação incident-response

O Problema

Monitorar é fácil. Saber o que fazer quando algo cai é que é difícil.

Todo time tem um playbook de incidentes no Notion/Confluence. Mas quando o alerta dispara às 3h da manhã, ninguém vai lá ler. O resultado? Respostas inconsistentes, atrasadas, ou simplesmente esquecidas.

Com o Mengão Monitor em v3.10 (Distributed Tracing), eu tinha visibilidade total. Mas faltava ação automática.

A Solução: Playbooks como Código

v3.11 introduz Incident Response Playbooks — playbooks executáveis que rodam automaticamente quando alertas são disparados.

Arquitetura

Alert Escalation (v3.2)
        ↓ trigger
Incident Playbook Manager (v3.11)
        ↓ match conditions
Playbook Actions
        ↓ execute
[LOG] [NOTIFY] [RUN_COMMAND] [HTTP_REQUEST] [CUSTOM]

7 Tipos de Ações

  • LOG — Apenas registra (com templates: {{endpoint}}, {{severity}})
  • NOTIFY — Envia notificação via webhook
  • RUN_COMMAND — Executa comando shell (com timeout e retry)
  • HTTP_REQUEST — Faz requisição HTTP (GET/POST/PUT/DELETE)
  • SCALE — Placeholder para escalonamento (K8s, Docker)
  • RESTART — Placeholder para restart de serviços
  • CUSTOM — Função Python registrada via register_handler()

Condições Flexíveis

conditions = [
    PlaybookCondition("endpoint", "eq", "API Principal"),
    PlaybookCondition("severity", "gte", 3),
    PlaybookCondition("response_time", "gt", 5000),
]

Operadores: eq, ne, gt, lt, gte, lte, contains, regex

Exemplo Real

Playbook para quando a API principal cai:

playbook = Playbook(
    id="api_principal_down",
    name="API Principal Down",
    trigger="alert_created",
    conditions=[
        PlaybookCondition("endpoint", "eq", "API Principal"),
        PlaybookCondition("severity", "gte", 2)
    ],
    actions=[
        PlaybookAction(
            action_type="log",
            name="Registrar incidente",
            config={"message": "Endpoint {{endpoint}} DOWN (status: {{status}})"}
        ),
        PlaybookAction(
            action_type="notify",
            name="Alertar time",
            config={"message": "🚨 {{endpoint}} está DOWN!"}
        ),
        PlaybookAction(
            action_type="run_command",
            name="Diagnóstico automático",
            config={"command": "curl -s -o /dev/null -w '%{http_code}' {{url}}"},
            retry_count=1
        )
    ],
    cooldown_seconds=600,  # 10 min entre execuções
    max_executions_per_hour=5
)

Proteções Built-in

  • Cooldown: Evita spam de execuções (configurável por playbook)
  • Rate Limiting: Max execuções por hora
  • Retry: Ações com retry automático e delay configurável
  • on_failure: continue, stop, ou rollback
  • Histórico: Todas as execuções registradas com detalhes

Templates Prontos
