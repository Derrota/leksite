Logs que Contam Histórias: O Poder do Logging Contextual

Todo mundo sabe que logs são importantes. Mas a maioria dos logs que eu vejo por aí são tipo diário de adolescente: "aconteceu uma coisa". Sem contexto, sem sequência, sem significado.

Depois de 38 versões do Mengão Monitor, aprendi que logs bons contam histórias. Não só o que aconteceu, mas por que aconteceu, quem estava envolvido, e qual era o estado do sistema naquele momento.

O Problema: Logs Sem Contexto

Veja este log típico:

2026-03-16 10:30:15 ERROR: Connection failed

Legal. Falhou. Mas falhou onde? Falhou para quem? Era a primeira tentativa ou a décima? O que o usuário estava fazendo?

Agora compare com logging contextual:

2026-03-16 10:30:15 ERROR: Connection failed
  request_id: abc123
  user_id: user_456
  endpoint: /api/v1/health
  attempt: 3/5
  last_error: "Connection refused"
  context: {
    "system_load": 0.85,
    "active_connections": 42,
    "memory_available": "512MB"
  }

Agora sim dá pra investigar. Dá pra correlacionar. Dá pra entender.

Os 5 Elementos de um Log com Contexto

1. Request ID (Correlation ID)

Cada request único recebe um ID que aparece em todos os logs relacionados. Quando algo dá errado, você busca pelo ID e vê toda a história daquela request.

def generate_request_id():
    return f"{int(time.time())}-{os.getpid()}-{random.randint(1000,9999)}"

2. User/Session Context

Quem estava fazendo o quê? Mesmo em sistemas internos, saber qual componente ou serviço iniciou a ação é crucial.

3. System State

CPU, memória, conexões ativas. Quando algo falha, o estado do sistema muitas vezes explica o porquê.

4. Attempt Count e Retry State

É a primeira tentativa ou a décima? Há diferença entre "falhou uma vez" e "falhou 10 vezes seguidas".

5. Business Context

O que o usuário estava tentando fazer? "Falhou ao processar pagamento" é diferente de "falhou ao carregar imagem do perfil".

Implementação Prática: 40 Linhas

Não precisa de framework. Não precisa de biblioteca de 50MB. Logging contextual em Python puro:

import json
import time
import os
from contextvars import ContextVar

# Contexto da request (thread-safe)
request_context: ContextVar[dict] = ContextVar('request_context', default={})

class ContextualLogger:
    def __init__(self, name: str):
        self.name = name
    
    def _log(self, level: str, message: str, **kwargs):
        ctx = request_context.get()
        log_entry = {
            "timestamp": time.strftime("%Y-%m-%dT%H:%M:%SZ", time.gmtime()),
            "level": level,
            "logger": self.name,
            "message": message,
            **ctx,
            **kwargs
        }
        print(json.dumps(log_entry, ensure_ascii=False))
    
    def info(self, msg: str, **kw): self._log("INFO", msg, **kw)
    def error(self, msg: str, **kw): self._log("ERROR", msg, **kw)
    def warn(self, msg: str, **kw): self._log("WARN", msg, **kw)

# Uso
logger = ContextualLogger("api.health")

# Setar contexto no início da request
request_context.set({
    "request_id": "abc123",
    "user_id": "user_456",
    "endpoint": "/api/v1/health"
})

# Todos os logs subsequentes têm contexto automático
logger.info("Starting health check", attempt=1)
logger.error("Connection failed", error="Connection refused", attempt=3)

Queries Úteis com jq

Com logs estruturados + contexto, você pode fazer queries que parecem mágica:

# Todos os logs de uma request específica
cat app.log | jq 'select(.request_id == "abc123")'

# Erros agrupados por usuário
cat app.log | jq -r 'select(.level == "ERROR") | .user_id' | sort | uniq -c

# Requests que falharam mais de 3 vezes
cat app.log | jq 'select(.attempt > 3 and .level == "ERROR")'

# Timeline de uma request (ordem cronológica)
cat app.log | jq -s 'sort_by(.timestamp) | .[] | select(.request_id == "abc123")'

Lições do Mengão Monitor

Em 38 versões, aprendi na marra:

  • Request ID salva vidas: Sempre que alguém reporta um bug, o primeiro pedido é "me dá o request_id"
  • Contexto demais é ruído: Inclua o essencial. Se você nunca usa um campo, remova
  • Timestamp ISO 8601: Sempre UTC. Sempre com timezone. Nunca confie em horário local
  • Não logue dados sensíveis: Senhas, tokens, CPFs nunca nos logs. Nunca
  • Log para investigação, não para auditoria: Se precisa de auditoria, use sistema separado

Conclusão

Logging contextual não é sobre ter mais logs. É sobre ter logs melhores. Logs que contam histórias completas, que permitem investigação rápida, que transformam "aconteceu uma coisa" em "aconteceu X porque Y, envolvendo Z, durante estado W".

A diferença entre debugar um problema em 5 minutos versus 5 horas muitas vezes está na qualidade dos logs. E qualidade vem de contexto.
