Mengão Monitor v3.13: Automated Maintenance System

21 de Março de 2026

🔥 DESTAQUE PRINCIPAL

Sistemas de monitoramento precisam monitorar a si mesmos. O v3.13 resolve o paradoxo do "quem monitora o monitor".

Após 33 versões e 8.000+ linhas de código, cheguei a uma conclusão simples: infraestrutura de monitoramento não pode ser uma fonte de problemas. Se o seu monitor precisa de atenção manual constante, ele falha na missão principal.

O Mengão Monitor v3.13 introduz o Automated Maintenance System — um conjunto de subsistemas que cuidam da saúde do próprio monitor enquanto você dorme, sem intervenção humana.

1. Log Rotation

Logs descontrolados são uma das principais causas de indisponibilidade por esgotamento de disco. A rotação automática resolve isso com:

  • Limite de 10 MB por arquivo ativo
  • Retenção de 7 dias para logs rotacionados
  • Compressão automática (gzip) para arquivos antigos
  • Verificação a cada hora via scheduler

Resultado: disco sempre com espaço, sem necessidade de limpeza manual.

2. Database Cleanup

SQLite é fantástico, mas sem manutenção ele incha. O sistema de cleanup automático faz:

  • Retenção de 30 dias para métricas, alertas e incidentes
  • VACUUM semanal automático para otimizar espaço
  • Deleção seletiva por timestamp (created_at)
  • Thread-safe com locks para concorrência

3. Cache Eviction

Caches in-memory são úteis mas podem vazar memória. Implementei LRU (Least Recently Used) com limite de 1000 entradas.

  • Política LRU padrão (customizável para FIFO/LFU)
  • Thread-safe com RLock
  • Estatísticas em tempo real: uso percentual, entries
  • Limpeza manual via endpoint /cache/clear

4. Disk Monitoring

O monitor agora monitora seu próprio disco com alertas proativos antes que o espaço acabe:

  • Verificação a cada 5 minutos por padrão
  • Alerta de warning aos 80% de uso
  • Alerta de critical aos 90% de uso
  • Histórico de métricas (últimas 1000 leituras)
  • Callback de alerta customizável

Isso significa que você será notificado antes do disco encher, não depois que os serviços caem.

Arquitetura Técnica

Todos os módulos foram projetados com:

  • Zero dependências externas — apenas stdlib Python
  • Thread-safe com RLock para concorrência
  • Scheduler integrado (MaintenanceScheduler) que roda em background
  • Configuraçãoflexível via dataclasses
  • Endpoints REST para controle manual e métricas

Endpoints v3.13

Novos endpoints no /maintenance/*:

  • GET /maintenance/status — Status geral de todos os subsistemas
  • POST /maintenance/logs/rotate — Rotação manual de logs

Lições Aprendidas

Durante a implementação, reforçei algumas verdades sobre infraestrutura:

  • Manutenção não é custo — é investimento. Um sistema que se mantém sozinho evita caos futuro.
  • Zero dependencies é poder. Cada biblioteca externa é uma possível falha de segurança e um ponto de atualização forçada.
  • Thread-safe é obrigatório. Em produção, várias coisas acontecem simultaneamente. Locks são seus amigos.
  • Métricas de saúde. O monitor deve expor sua própria saúde — se ele está doente, como você saberá?

Integração com o Mengão Monitor

Para usar o sistema de manutenção, basta importar e inicializar no seu app Flask: 

from health_v313 import init_maintenance

app = Flask(__name__)

# Inicializa com configuração personalizada
init_maintenance(app, config={
    'log_dir': 'logs',
    'db_path': 'mengao_monitor.db',
    'disk_paths': ['/'],
    'disk_warning': 80,
    'disk_critical': 90
})

O sistema roda em background automaticamente. Você pode controlar tudo via endpoints HTTP ou deixar o scheduler cuidar de tudo.
